# 프론트엔드 잔여 이슈 해결(1차) 결과 보고서

본 문서는 `stockGame_react`의 설정 최적화 및 메모리 누수 방지 작업 결과를 정리한 문서입니다.

## 🛠️ 수정 사항 요약

### 1. `vite.config.js` 프록시 서버 설정 도입
- **이전**: 프론트엔드 코드 내(`axiosConfig.js`, `SockJS`)에 백엔드 주소(`http://localhost:8882`)가 모두 하드코딩 되어있었습니다.
- **변경 후**: Vite 개발 서버(`5173`) 단에서 `/api`와 `/ws` 경로로 들어오는 요청을 가로채 백엔드로 전달하도록 프록시(`proxy`)를 설정했습니다.
- **효과**: 
  - 프론트엔드 코드 내부에서는 더 이상 도메인 주소를 알 필요 없이 `/api/...` 형태의 상대 경로로 깔끔하게 요청을 보낼 수 있습니다.
  - 브라우저 입장에서는 프론트엔드와 API 서버의 출처(Origin)가 같다고 인식하게 되어 잠재적인 CORS 문제를 원천 차단합니다. 배포 환경(Nginx 등)의 설정 방식과 동일한 아키텍처를 가집니다.

### 2. `StockDetail.jsx` WebSocket 메모리/연결 누수 수정
- **이전**: 종목 상세 페이지에 들어갈 때마다 웹소켓에 연결하고, 페이지를 벗어날 때 연결을 끊어주는 로직이 비어있었습니다. 사용자가 뒤로가기를 반복하면 웹소켓 커넥션이 폭발적으로 누적되는(좀비 커넥션) 치명적인 결함이 있었습니다.
- **변경 후**: `connectWebSocket` 함수가 생성된 `stompClient` 객체를 리턴하게 만들고, `useEffect`의 `return () => {}` (Cleanup 함수)에서 `stompClient.deactivate()`를 명시적으로 호출하게 하였습니다.
- **효과**: 사용자가 다른 페이지로 이동하거나 컴포넌트가 파괴될 때, 기존 웹소켓 연결을 완벽하게 끊고 리소스를 반환하여 안정성을 확보했습니다.

## 🚀 검증 및 결론
- 로컬 DB 서버와의 통합 테스트는 추후 프론트 구동 시 진행해야 하지만, 문법과 구조적 리팩토링은 모두 성공적으로 반영되었습니다.
- 추후 백엔드 서버를 켜고 `npm run dev` 시 프록시 라우팅과 웹소켓 토스트 메시지가 완벽히 작동할 준비가 되었습니다.
